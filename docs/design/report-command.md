# Report Command — Implementation Plan

## Context

The `analyze` command produces data — tables, CSV, JSON — optimized for human scanning or machine consumption. But a growing use case is embedding hotspot context into markdown documents read by AI coding agents (`CLAUDE.md`, `agents.md`, `CONTRIBUTING.md`). These agents benefit from structured prose: what the hotspots are, what the quadrants mean, and how to approach files in each category.

The `report` command is a pure presentation layer. It consumes the JSON output of `analyze` and renders structured markdown. It performs no analysis itself — it transforms existing results into a format designed for embedding in agent-readable documents.

---

## How It Works

`report` reads a JSON array of file scores (the output of `hc analyze --format json`) from stdin or a file, groups entries by quadrant, and renders a markdown document with sentinel markers for idempotent upsert into existing files.

```bash
# Pipe from analyze
hc analyze --since "6 months" --format json | hc report

# Read from file
hc report --input results.json

# Upsert into an existing markdown file
hc analyze --since "6 months" --format json | hc report --output CLAUDE.md
```

### Output structure

```markdown
<!-- hc:report:start -->
## Codebase Hotspot Analysis

This analysis classifies files into a 2×2 matrix of **churn** (commit frequency)
× **complexity** (structural size), identifying where development effort concentrates
and where risk accumulates. Complexity metric: **loc**.

### Critical Hotspots — High Churn, High Complexity

These files are both complex and actively changing. They carry the highest risk of
defects and are the most expensive to modify. Prioritize test coverage, small
focused PRs, and careful review when working in these files.

| Path | Commits | Lines | Complexity | Authors |
|------|---------|-------|------------|---------|
| src/parser.go | 87 | 1240 | 1240 | 6 |

### Hot but Simple — High Churn, Low Complexity

Frequently changed but structurally simple. These are typically configuration,
glue code, or generated files. Low risk — changes here are unlikely to introduce
subtle bugs.

| Path | Commits | Lines | Complexity | Authors |
|------|---------|-------|------------|---------|
| src/handlers.go | 45 | 120 | 120 | 3 |

### Cold & Complex — Low Churn, High Complexity

Complex files that are rarely modified. They are stable liabilities — safe today,
but risky to change because they are poorly understood. Add tests and documentation
before modifying. Do not refactor proactively.

| Path | Commits | Lines | Complexity | Authors |
|------|---------|-------|------------|---------|

### Cold & Simple — Low Churn, Low Complexity

Stable and simple. These files require no special attention.

| Path | Commits | Lines | Complexity | Authors |
|------|---------|-------|------------|---------|

*Generated by `hc report` on 2026-03-29.*
<!-- hc:report:end -->
```

### Sentinel markers

The start and end markers are HTML comments, invisible in rendered markdown:

```
<!-- hc:report:start -->
<!-- hc:report:end -->
```

These enable three operations:

- **Detect**: scan for `<!-- hc:report:start -->` — if present, the section exists.
- **Update**: replace everything from start marker through end marker (inclusive) with fresh output.
- **Insert**: if no markers found, append to end of file.

The markers are stable strings — they contain no metadata, timestamps, or flags. Metadata lives inside the markers as rendered prose (the timestamp line, the metric label), so detection logic is a simple string match.

## Points of Integration

### JSON input schema

`report` consumes the existing `fileJSON` structure from `internal/output`:

```json
[
  {
    "path": "src/parser.go",
    "commits": 87,
    "lines": 1240,
    "complexity": 1240,
    "authors": 6,
    "quadrant": "hot-critical",
    "metric": "loc"
  }
]
```

The `quadrant` field (kebab-case string) determines grouping. The `metric` field is read from the first entry to label the complexity column in the output. All fields are included in the markdown table.

`report` also accepts the `dirJSON` structure (detected by the presence of `files` and `total_commits` keys). When directory-level JSON is provided, the table columns adjust accordingly.

### New package: `internal/report`

```
internal/report/
    report.go       — JSON parsing, quadrant grouping, markdown rendering
    upsert.go       — file reading, marker detection, section replacement
    report_test.go
    upsert_test.go
```

**`report.go`**

```go
// Render reads JSON from r, groups by quadrant, and writes structured markdown to w.
func Render(r io.Reader, w io.Writer) error
```

Internally:

1. Decode JSON array into `[]fileJSON` or `[]dirJSON` (try file first, fall back to dir based on field presence).
2. Group entries by `quadrant` field.
3. Render each quadrant section with its description and table.
4. Wrap output in sentinel markers.

**`upsert.go`**

```go
const (
    MarkerStart = "<!-- hc:report:start -->"
    MarkerEnd   = "<!-- hc:report:end -->"
)

// UpsertFile reads the file at path, replaces the report section if it exists
// (or appends if not), and writes the file back. The content should include markers.
func UpsertFile(path string, content string) error

// HasSection returns true if the file contains the start marker.
func HasSection(data []byte) bool

// ReplaceSection replaces everything between start and end markers (inclusive)
// with the new content. Returns the original data unchanged if markers are not found.
func ReplaceSection(data []byte, content string) []byte
```

`UpsertFile` orchestrates the flow:

1. Read the existing file (or treat as empty if it doesn't exist).
2. If `HasSection` is true, call `ReplaceSection`.
3. Otherwise, append `"\n" + content` to the end.
4. Write the file back.

### CLI (`cmd/hc/main.go`)

Add a `report` subcommand alongside `analyze`:

```go
{
    Name:  "report",
    Usage: "Render analysis JSON as markdown for embedding in agent docs",
    Flags: []cli.Flag{
        &cli.StringFlag{
            Name:  "input",
            Usage: "Path to JSON file (default: stdin)",
        },
        &cli.StringFlag{
            Name:  "output",
            Usage: "Markdown file to upsert into (default: stdout)",
        },
    },
    Action: runReport,
}
```

`runReport` logic:

1. Open input source: if `--input` is set, open the file; otherwise use `os.Stdin`.
2. Call `report.Render(input, &buf)` to produce markdown.
3. If `--output` is set, call `report.UpsertFile(outputPath, buf.String())`.
4. Otherwise, write `buf` to `os.Stdout`.

### Interaction with existing packages

- `internal/output` — no changes. `report` consumes the JSON it already produces; the `fileJSON` and `dirJSON` types define the contract but `report` decodes into its own local struct (to avoid an import dependency on `output`, which imports `analysis`).
- `internal/analysis` — no changes. The quadrant kebab-case strings (`hot-critical`, `hot-simple`, etc.) are the stable interface between `analyze` and `report`.
- `internal/complexity`, `internal/git` — no interaction. `report` has no knowledge of the analysis pipeline.

## Flag Considerations

### `--input`

- Optional. Defaults to stdin, enabling pipe composition with `analyze`.
- Accepts a path to a JSON file for workflows where the analysis is saved and the report is generated later.

### `--output`

- Optional. Defaults to stdout.
- When set, performs the upsert operation: read file → detect markers → replace or append → write file.
- Creates the file if it doesn't exist (writes the full marked section).
- The file is written atomically (write to temp, rename) to avoid corruption on failure.

### Why no `--template` flag yet?

The initial implementation produces one format. If demand appears for variants (e.g. a more concise version, a version with different section headings for `CONTRIBUTING.md` vs `CLAUDE.md`), a `--template` flag can be added later. The rendering logic in `report.go` is structured to make this easy — quadrant grouping and table rendering are separate from section prose — but the flag isn't worth adding until there are two templates.

## Risks

### JSON schema drift

If `fileJSON` or `dirJSON` fields are renamed or removed in `internal/output`, `report` will silently produce wrong results or fail to parse.

**Mitigation:** `report` decodes into its own local structs and validates required fields (`path`, `quadrant` are mandatory; others default to zero). Add a test that round-trips through `output.FormatFiles` → `report.Render` to catch schema drift.

### Large JSON input

A monorepo with tens of thousands of files produces a large JSON array. Rendering all of them into markdown tables would produce an unreadable document.

**Mitigation:** `report` renders at most 15 entries per quadrant by default, with a note like "(and 342 more)" when truncated. The input should already be limited via `--top` on the `analyze` side, but `report` defends against unbounded input regardless.

### Upsert file permissions and encoding

Writing back to a file the user didn't expect to modify (e.g. a read-only `CLAUDE.md`) would fail or corrupt.

**Mitigation:** `UpsertFile` checks write permission before modifying. Uses temp file + rename for atomic writes. Preserves the original file's trailing newline convention.

### Marker collision

Another tool could theoretically use `<!-- hc:report:start -->` as a marker.

**Mitigation:** The `hc:report:` prefix is specific enough. This is a negligible risk.

### stdin detection

When the user runs `hc report` without piping input and without `--input`, the command blocks waiting on stdin with no feedback.

**Mitigation:** If stdin is a terminal (not a pipe), print a usage hint to stderr: `reading JSON from stdin... (use --input or pipe from "hc analyze --format json")`.
